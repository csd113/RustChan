## Overview

RustChan is a fully-featured imageboard compiled into a single Rust binary. Drop it on a VPS, a Raspberry Pi, or a local machine and it runs immediately — no containers, no runtime, no package manager required. All data lives in a single directory next to the binary, making backups and migrations trivial.

### Feature Highlights

| Category | Details |
|---|---|
| **Boards** | Multiple boards with per-board SFW / NSFW tagging, bump limits, and thread limits |
| **Posts** | Threaded replies, tripcodes, greentext markup, quote-linking, subject lines |
| **Media** | Image uploads (JPEG, PNG, GIF, WebP) and video uploads (MP4, WebM) with auto-generated thumbnails |
| **Views** | Board index with reply previews, catalog grid view, per-board full-text search, pagination |
| **Admin** | Web panel for board management, post deletion, thread sticky/lock, IP banning, word filters |
| **Security** | Argon2id passwords, CSRF protection, IP-hash-based rate limiting, `HttpOnly` session cookies |
| **Themes** | Five built-in UI themes, user-selectable and persisted per browser |
| **Deployment** | Systemd service file, nginx reverse proxy config, and automated backup script included |

## Quick Start

```bash
# 1. Build
cargo build --release

# 2. Create your first admin account
./chan admin create-admin admin "YourPassword123!"

# 3. Create some boards
./chan admin create-board b    "Random"     "General discussion"
./chan admin create-board tech "Technology" "Programming and hardware"

# 4. Run
./chan
```

Open **http://localhost:8080** in your browser. The admin panel is at **/admin**.

On first launch, `chan-data/settings.toml` is generated next to the binary with all configurable values and comments. Edit it and restart to apply changes.

## Data Layout

All persistent data is stored in `chan-data/` alongside the binary. No files are written elsewhere unless you override the paths via environment variables.

```
chan                          ← single self-contained binary
chan-data/
├── settings.toml             ← instance configuration (auto-generated on first run)
├── chan.db                   ← SQLite database (WAL mode)
└── uploads/
    ├── <uuid>.<ext>          ← original uploaded files
    └── thumbs/
        └── <uuid>_thumb.jpg  ← auto-generated thumbnails
```

## Configuration

### settings.toml

Generated automatically on first run with a randomly-generated `cookie_secret`. Edit the file and restart the server for changes to take effect.

```toml
# RustChan — Instance Settings

# Name shown in the browser title, header, and home page.
forum_name = "RustChan"

# Port to listen on (binds to 0.0.0.0:<port>).
port = 8080

# Maximum upload size in megabytes for images (JPEG, PNG, GIF, WebP).
max_image_size_mb = 8

# Maximum upload size in megabytes for video (MP4, WebM).
max_video_size_mb = 50

# Secret used for IP hashing and CSRF tokens.
# Auto-generated on first run. Do NOT change after your instance has posts —
# all existing IP hashes (and therefore bans) will become invalid.
cookie_secret = "<auto-generated>"
```

### Environment Variables

All settings can be overridden with environment variables. Environment variables take precedence over `settings.toml`. This is recommended for secrets in production (e.g. via systemd's `Environment=` directive).

| Variable | Default | Description |
|---|---|---|
| `CHAN_FORUM_NAME` | `RustChan` | Site display name |
| `CHAN_PORT` | `8080` | TCP port (used only if `CHAN_BIND` is not set) |
| `CHAN_BIND` | `0.0.0.0:8080` | Full bind address (overrides `CHAN_PORT`) |
| `CHAN_DB` | `<exe-dir>/chan-data/chan.db` | Path to the SQLite database file |
| `CHAN_UPLOADS` | `<exe-dir>/chan-data/uploads` | Path to the uploads directory |
| `CHAN_COOKIE_SECRET` | *(from settings.toml)* | **Set this in production.** Used for CSRF tokens and IP hashing. |
| `CHAN_MAX_IMAGE_MB` | `8` | Maximum image upload size in MiB |
| `CHAN_MAX_VIDEO_MB` | `50` | Maximum video upload size in MiB |
| `CHAN_THUMB_SIZE` | `250` | Thumbnail maximum dimension in pixels |
| `CHAN_BUMP_LIMIT` | `500` | Reply count after which a thread stops bumping |
| `CHAN_MAX_THREADS` | `150` | Maximum threads per board before the oldest is pruned |
| `CHAN_RATE_POSTS` | `10` | Maximum POSTs per rate-limit window per IP |
| `CHAN_RATE_WINDOW` | `60` | Rate-limit window duration in seconds |
| `CHAN_SESSION_SECS` | `28800` | Admin session duration in seconds (default: 8 hours) |
| `CHAN_BEHIND_PROXY` | `false` | Set `true` when running behind nginx or Caddy to trust `X-Forwarded-For` |
| `CHAN_HTTPS_COOKIES` | *(same as `CHAN_BEHIND_PROXY`)* | Set `true` to add `Secure` flag to cookies |
| `RUST_LOG` | `chan=info` | Log verbosity (`chan=debug` for verbose output) |

## Admin CLI

Board and account management is performed through the built-in CLI subcommand. No separate tool is needed.

```bash
# Account management
chan admin create-admin   <username> <password>
chan admin reset-password <username> <new-password>
chan admin list-admins

# Board management
chan admin create-board <short> <name> [description] [--nsfw]
chan admin delete-board <short>
chan admin list-boards

# Ban management
chan admin ban       <ip_hash> "<reason>" [duration_hours]  # omit hours = permanent
chan admin unban     <ban_id>
chan admin list-bans
```

`<short>` is the board slug used in URLs (e.g. `tech` → `/tech/`). Keep it lowercase, 1–8 characters.

## Themes

RustChan ships with five selectable UI themes. Users access the theme picker via the fixed button in the bottom-right corner of every page. The selection is saved to `localStorage` and applied on subsequent visits with no flash.

## Security Model

| Concern | Implementation |
|---|---|
| **Password hashing** | Argon2id (`t=2, m=65536, p=2`) — memory-hard, GPU-resistant. ~200ms per hash on a Raspberry Pi 4. |
| **Session cookies** | `HttpOnly`, `SameSite=Strict`, path-scoped to `/admin`. Duration configurable (default 8h). |
| **CSRF protection** | Double-submit cookie pattern. Every POST form includes a `_csrf` field validated against the session cookie. Cross-origin requests cannot forge valid submissions. |
| **IP privacy** | Raw IP addresses are never stored. A salted SHA-256 hash keyed to `cookie_secret` is stored instead — the salt prevents rainbow table attacks against a leaked database. |
| **Rate limiting** | In-memory sliding window (per hashed IP) using `DashMap`. Default: 10 POSTs per 60 seconds. |
| **File validation** | Two-layer check: MIME type from the `Content-Type` header, and magic byte inspection of the file content itself. Extension is never trusted. |
| **XSS prevention** | All user input is HTML-escaped before rendering. Post markup (greentext, reply links) is applied after escaping — no raw user HTML ever reaches the browser. |

## Architecture

RustChan is intentionally minimal. There is no template engine, no ORM, and no JavaScript framework. HTML is rendered with plain Rust string formatting. The result is a single binary of approximately 12–18 MiB that starts in under a second.

| Component | Technology |
|---|---|
| Web framework | [Axum](https://github.com/tokio-rs/axum) 0.7 |
| Async runtime | [Tokio](https://tokio.rs/) 1.x |
| Database | SQLite via [rusqlite](https://github.com/rusqlite/rusqlite) (bundled — no system library needed) |
| Connection pool | r2d2 + r2d2_sqlite |
| Image processing | [image](https://github.com/image-rs/image) crate (JPEG, PNG, GIF, WebP) |
| Password hashing | argon2 crate (Argon2id) |
| HTML rendering | Plain Rust `format!` strings — no template engine |
| Configuration | `settings.toml` + environment variable overrides via `once_cell::Lazy` |
| Logging | `tracing` + `tracing-subscriber` → stdout / journald |

### Source Layout

```
src/
├── main.rs            — server entry point, router, background stats task
├── lib.rs             — library root (shared between server and CLI)
├── config.rs          — configuration loading (env vars → settings.toml → defaults)
├── db.rs              — all SQL queries (no ORM)
├── error.rs           — unified AppError type with HTTP response conversion
├── models.rs          — data structs mirroring database rows
├── middleware/
│   └── mod.rs         — rate limiting, CSRF extraction, IP hashing, proxy support
├── handlers/
│   ├── board.rs       — board index, thread creation, search, user post deletion
│   ├── thread.rs      — thread view, reply posting
│   └── admin.rs       — admin panel, board/ban/filter management
├── templates/
│   └── mod.rs         — pure-Rust HTML generation (all five themes, theme picker JS)
└── utils/
    ├── crypto.rs      — Argon2id, CSRF tokens, IP hashing
    ├── files.rs       — upload validation, thumbnail generation
    ├── sanitize.rs    — HTML escaping, post markup (greentext, quote-links)
    └── tripcode.rs    — SHA-256 tripcode system

deploy/
├── chan.service       — systemd unit file
├── nginx.conf         — reverse proxy configuration
└── backup.sh          — SQLite hot-backup script

static/
└── style.css          — all styles: base theme + four theme overrides + theme picker widget
```

## Building

### Native (recommended for the target machine)

```bash
cargo build --release
# Binary: target/release/chan
```

### Cross-compilation to ARM64 (Raspberry Pi 4/5)

```bash
rustup target add aarch64-unknown-linux-gnu
cargo install cross   # uses Docker; handles the cross-linker automatically
cross build --release --target aarch64-unknown-linux-gnu
# Binary: target/aarch64-unknown-linux-gnu/release/chan
```

### Strip and size

The release profile (`Cargo.toml`) sets `strip = true`, `lto = "thin"`, and `panic = "abort"`. Typical stripped binary size is 12–18 MiB.
